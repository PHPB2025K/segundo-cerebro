# Sessão — 2026-04-30 (noite — fix urgente ML) — IA respondendo evasivamente

`process-ml-question` JÁ chamava `search_corrections` RPC corretamente, mas com `match_threshold=0.85`. Perguntas de buyers no ML variam muito em forma:

- Pergunta nova: *"podem ir ao forno, microondas, congelador?"*

Semanticamente equivalentes, mas similaridade de embedding fica ~0.7. RPC retornava vazio → contexto pra Claude sem correção → Claude caía no fallback evasivo seguro.

13 correções na tabela, todas com embedding gerado. Não era problema de pipeline de embeddings.

## Fix #1 — Threshold + prompt (commit `6d03759`)

- Threshold `0.85 → 0.65` em 2 lugares:
  - `process-ml-question/index.ts` (perguntas Mercado Livre)
  - `_shared/context-builder.ts` (mensagens WhatsApp)
- Bloco de correção no prompt reescrito com tom imperativo: *"RESPOSTAS APROVADAS PARA PERGUNTAS SIMILARES (USE OBRIGATORIAMENTE)"* + *"NUNCA responda 'entre em contato conosco' se há uma resposta aprovada"* + similaridade % visível por linha.

Reduz drasticamente a probabilidade da resposta evasiva, mas não garante (LLM ainda pode escapar do prompt).

## Fix #2 — Hard-block determinístico (commit `1b88990`)

Pedro pediu garantia absoluta: *"vc garantiu que a [Ana] nunca MAIS Responda enviando nosso contato?"* O fix #1 era probabilístico — para garantir, precisava validador determinístico após geração.

- Novo `detectForbiddenContactRequest()` em `_shared/ml-response-validator.ts` — 10 padrões regex:
  - `entre em contato`, `entrar em contato`, `fale conosco`
  - `nos contate / chame / procure / envie / mande`
  - `me chame / envie / mande / contate`
  - `contate-nos / contate nosso`
  - `envie / mande mensagem ou email`
  - `estamos à disposição`
  - `para mais detalhes ... contat / fal / chame`
- Integração em `validateMLResponse`: quando detectado, retorna `text=''` + flag `forbiddenContactDetected=true`, forçando substituição.
- Em `process-ml-question`, cadeia de substituição:
  1. Top-similarity correção (`search_corrections` com threshold 0.55 — mais permissivo aqui pois sabemos que LLM falhou)
  2. Fallback técnico neutro: *"Para essa especificação técnica do produto, recomendo verificar a descrição completa do anúncio. Caso a informação não esteja lá, podemos atualizá-la."*
  3. Hardcoded last-resort que validator não pode rejeitar.
- Logs estruturados: `forbidden_contact_blocked` (com question_id + reasons + texto original) + `forbidden_contact_substituted_by_correction` ou `_by_fallback` pra auditoria.

- **Threshold 0.85 é alto demais para perguntas reais de buyers em marketplace.** 0.65 captura reformulações; mais que isso seria seguro pra correção pura mas perdíamos demais. Reavaliação periódica conforme volume cresce.
- **Prompt forte != garantia.** LLMs ignoram instruções em casos extremos. Para padrões de risco regulatório (compliance ML, vazamento de dados), validação determinística pós-geração é obrigatória — não delegar ao prompt.
- **Cadeia de substituição com fallback decrescente** é mais robusta que tentar reescrever a resposta. Saber quando admitir derrota e usar texto neutro evita pior caso (resposta inventada que viola compliance).
