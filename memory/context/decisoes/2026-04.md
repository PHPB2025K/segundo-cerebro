## [13/04] Amazon Request Review — Fix definitivo (campo de data + modelo)

**Contexto:** CRON "Amazon Request Review Diário" nunca funcionou corretamente desde 01/04. Taxa de sucesso real: 2-3%. Dois problemas independentes:
1. Script filtrava por `order_date` (data de compra) mas Amazon calcula janela 5-30 dias a partir da **data de entrega** → 97% das solicitações rejeitadas com 403 "outside window"
2. Modelo GPT-5.4 dava timeout em ~50% das execuções. Fallback Claude Sonnet quebrado por mudança de billing Anthropic.
**Decisões:**
- **DB:** Adicionada coluna `delivered_at TIMESTAMPTZ` + índice na tabela `orders`. Script de sync alterado para popular campo (prioridade: LatestDeliveryDate > EarliestDeliveryDate > LatestShipDate). Script de reviews agora filtra por `delivered_at`.
- **Modelo:** Cron migrado de `openai-codex/gpt-5.4` para `anthropic/claude-opus-4-6` (exclusivo, sem fallback GPT). Timeout aumentado de 420s para 600s.
- **Infra:** Log rotation com timestamp + alertas automáticos (CRITICAL se 0 sucesso >10 elegíveis, WARNING se rejeição >50%).
- **Backfill:** Script criado e executado para popular `delivered_at` em ~4866 pedidos existentes. Piloto 100 orders: 100% fill rate, datas saudáveis (1-2 dias FBA).
**Nota:** SP-API BR não retorna `LatestDeliveryDate`/`EarliestDeliveryDate` — usa `LatestShipDate` como proxy (entrega FBA = ship + 1-5 dias úteis).
**Validação:** 10/10 smoke tests PASS. Backfill completo lançado (~80 min estimados).
**Owner:** Pedro (via Claude Code + SSH, 3 subagentes Opus 4.6 em paralelo)
**Status:** ✅ Fix implementado + backfill em execução
