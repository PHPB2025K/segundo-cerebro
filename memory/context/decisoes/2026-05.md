### Decisões fundadoras

1. **Padrão SPEC-first.** Todo módulo do Mission Control passa a ter SPEC estruturada (Visão / Estado atual vs alvo / Layout / APIs / Componentes / Jornadas / Flows / Estados / Critérios / Out-of-scope / Ordem de implementação / Decisões pendentes) antes de qualquer código. Pedro confirma 4-5 decisões pendentes, daí começo. Estabelecido em 12/05 (Dashboard) e reforçado em 13/05 (System + Files).

2. **Vertical slices ponta-a-ponta.** Cada slice = backend + frontend + integração + validação, deployado e testado antes da próxima. Não fazer waterfall (toda API → toda UI). Pedro confirmou "Vertical Slices" na decisão do Dashboard.

3. **Server Components por padrão.** Apenas Client islands onde precisa interatividade/animação (Notepad auto-save, StatusDot pulse, CommandPalette, AgentLive SSE, Sparkline animado, etc.). `fetchInternal` helper repassa cookies httpOnly pra Server Components não baterem 401 no middleware.

4. **Persistência via SQLite onde fizer sentido.** Padrão Mission Control: cada módulo pode ter sua tabela em DB próprio ou compartilhar `activities.db`. Decisões aplicadas: Notepad single-row em `activities.db`, métricas Hardware em `system-metrics.db` 24h retention, lixeira de Files em `data/trash/<ws>/`.

5. **Anti-foot-gun obrigatório em ações destrutivas.** `mission-control` não pode stop via /system (só restart). `evolution-postgres` requer `x-confirm: yes` header. Delete em /files exige digitar o nome do arquivo. Allowlist runtime em vez de hardcoded.

6. **Decisão de escopo /system vs futuro "Minhas Aplicações".** `/system` é puro saúde da VPS (PM2 + Docker + Gateway + métricas Linux). Sistemas cross-host (Vercel, Supabase, Railway) viram **PRD futuro próprio** "Minhas Aplicações" — não misturar conceitualmente. GB Import Hub (que roda via nginx+systemd) **fica de fora do /system** apesar de detectável, vai para o PRD futuro.

## [07/05] Nginx host da VPS — `enable` ativado (era `disabled` desde sempre)

**Contexto:** Pedro relatou `https://import.budamix.com.br/` em branco. Diagnóstico: HTTP 502 do Traefik. Causa raiz: nginx host estava `inactive` + `disabled` desde o reboot da VPS em 30/04/2026 16:59 (7 dias offline). PM2 e Docker voltaram automaticamente (têm autostart configurado), mas nginx host nunca subiu.

**Arquitetura confirmada:** Traefik (Docker, :80/:443, faz SSL) → roteia hosts estáticos para nginx host na :8090 → nginx serve SPA build de `/var/www/<vhost>/`. Vhosts afetados: `gb-import-hub` (import.budamix.com.br), `reports` (openclaw-uk7d.srv1480018.hstgr.cloud), `shopee-oauth` (api.importadoragb.com.br). Os 3 estavam fora há 7 dias.

**Decisão:** `systemctl start nginx && systemctl enable nginx`. Os 3 vhosts ressuscitaram juntos (validado via curl 200). `enable` cria o symlink em `multi-user.target.wants` para garantir auto-start em reboots futuros — caso recorrente: já tinha acontecido antes do Pedro perceber, agora não acontece de novo.

**Lição:** sempre que adicionar app no servidor que não roda em Docker/PM2, validar que o systemd unit está `enabled`. PM2 e Docker têm autostart por padrão, systemd unit não.

## [07/05] Telegram debounce inbound finalmente ativo (8s)

**Contexto:** Pedro mandou "Fala Kobe" e "Bom dia" praticamente no mesmo segundo e Kobe respondeu cada uma como turn separado. SOUL.md falava "Debounce 8s" mas era doc desatualizada — nunca esteve ativo no canal Telegram.

**Causa raiz:** `openclaw.json` tinha `messages.queue.debounceMs: 8000` (fila de SAÍDA, mode=collect, cap=20) confundido com debounce de inbound. O resolver real é `messages.inbound.debounceMs` (e `messages.inbound.byChannel.telegram` como override por canal) — chave que não existia. Telegram chamava `resolveInboundDebounceMs({cfg, channel: "telegram"})` em `extensions/telegram/bot-B7D5mOkP.js` e caía no fallback `?? 0`. Cada mensagem disparava turn imediato.

**Decisão:** adicionado bloco `messages.inbound: { debounceMs: 8000, byChannel: { telegram: 8000 } }` ao `openclaw.json`. Mensagens chegando em até 8s são concatenadas e processadas como um turn só. Comandos explícitos (`/new`, `/cerebro`) bypassam via `hasControlCommand`. Backup `openclaw.json.bak.20260507_115257` salvo antes da edição.

**Trade-off:** mensagem isolada agora espera 8s antes do Kobe começar a pensar. Aceitável — alternativa era continuar tendo turns duplicados.

**Aplicado:** gateway reiniciado, JSON validado, listener ativo (PID 3573404).
