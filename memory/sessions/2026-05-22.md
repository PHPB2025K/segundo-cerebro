## Manhã (12:08 BRT) — Loop de estoque site ↔ planilha (fix do oversell potencial)

### O que foi feito
- **Identificado problema raiz**: cada venda no site só decrementava `product_variants.stock` no Supabase via `decrement_stock` RPC chamada do `mp-webhook`. A aba ESTOQUE da planilha (fonte de verdade) não sabia da venda. O cron existente `sync.py` (a cada 5min) lia a planilha intacta e reescrevia o stock antigo — apagando o decremento.
- **Solução implementada**: novo cron VPS reverso (planilha ← site) que executa a cada 2 minutos.
  - Coluna `orders.planilha_decremented BOOLEAN DEFAULT false` adicionada + índice parcial `WHERE status='paid' AND planilha_decremented=false`.
  - Script `/opt/budamix-stock-sync/decrement-planilha.py` lê pedidos pendentes, faz lookup na aba SITE (SKU ANUNCIO → SKU BASE + UND POR KIT, suporta compostos via lista separada por vírgula), acumula decremento por SKU base, escreve via `gog sheets update --values-json` na col A da aba ESTOQUE, marca order como processada.
  - Wrapper `/root/scripts/decrement-planilha-cron.sh` com `GOG_KEYRING_PASSWORD` + `GOG_ACCOUNT=gb.ai.agent@gbimportadora.com`.
  - Crontab `*/2 * * * * /root/scripts/decrement-planilha-cron.sh` ativo.
  - Log: `/var/log/budamix-decrement-planilha.log`.
- **Bug menor descoberto e corrigido**: `gog sheets update` usa `--values-json` (2D array em JSON), não `--values`.
- **Teste end-to-end validado**:
  - Pedido fake: 1 kit Tulipa Preta (1 × TL250P × und=6 = 6 unidades).
  - 12:10:44 — planilha L71 565 → 559 (−6) ✅
  - 12:11:00 — sync.py propagou: 559/6 = 93 kits no Supabase ✅
  - 12:11:28 — ciclo seguinte estável (não voltou pra 94) ✅
  - Pedido removido + estoque restaurado.

### Decisões
- **Cron reverso na VPS via `gog` CLI** (em vez de webhook direto na edge function Deno chamando Sheets REST). Razão: `gog` já configurado, credenciais existem, Python familiar vs Deno + service account novo. Reaproveita infra.
- **Planilha continua SSoT do estoque físico** — venda no site vira input na planilha. Site nunca é fonte; sempre leitor/escritor. Arquitetura escala pra qualquer canal futuro.
- **Latência ≤7min aceita** (decrement 2min + sync.py 5min) vs trigger AFTER UPDATE no banco (latência zero). Trade-off OK pra volume atual; trigger fica nice-to-have, não bloqueante.
