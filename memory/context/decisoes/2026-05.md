## [05/05 noite] Social Studio PR2 — C4 conservador, C5 deploy, C6 restore zero-custo, escopo PR3

**Contexto:** sessão da tarde/noite de 05/05 fechou C4, C5 e C6 do PR2 do Social Studio (`budamix-ecommerce`). Validado visualmente pelo Pedro. Decisões críticas tomadas durante a implementação:

**1. C4 — padrão conservador `colorOverrideForSlot` (não `resolveSlotColor`).**

`resolveSlotColor` (definido no C2) tem precedência `custom → default semântico → fallback do caller`. Isso muda visual quando `SLOT_DEFAULT_TOKEN` diverge do que o template renderiza hoje (ex: `number: 'palette_title'` mas CoverNumeric renderiza `palette.accents.highlight`; `eyebrow` no Eyebrow component tem regra dinâmica `isClara → title : eyebrow`).

Decisão: criar helper alternativo `colorOverrideForSlot()` que retorna `undefined` se sem custom — caller faz `?? palette.X` direto. Visual 100% preservado quando sem customização. `SLOT_DEFAULT_TOKEN` continua válido só pra UX do ColorTokenPicker (destacar swatch "ativo").

Razão: Pedro quer "validar visualmente depois" e não quer que o C4 mude carrosséis em produção sem opt-in explícito. Conservadorismo > consistência semântica neste momento.

**2. C5 — `prompt = imageBrief` (não prompt completo enviado ao Gemini).**

Edge function `social-generate-image` faz `buildPrompt(imageBrief, aiKeywords)` que concatena brief + ai_keywords da paleta + RESTRICTIONS. Mas no entry de `image_versions`, salva apenas `imageBrief` original. Coerência com `slide.image_prompt` (que já guarda só o brief) e o backfill C1.5. `aiKeywords` e `RESTRICTIONS` são uniformes; o brief é o que diferencia gerações.

**3. C6 — restaurar versão é zero-custo, não bumpa `image_version` numérica.**

Restore só aponta `image_asset_id` de volta + `render_hash=null` + `render_status=pending` + `image_url=null` (força re-resolve signed URL). NÃO toca `image_versions` (array preservado), NÃO bumpa `image_version` (numérico monotônico — restore aponta de volta, não cria nova entry), NÃO adiciona custo em `ai_cost_usd`. A "versão ativa" exibida na badge é derivada do índice da entry cujo `image_asset_id == slide.image_asset_id`.

**4. PR3 escopo expandido — módulo coeso "controle manual de imagens".**

Antes: PR3 era modo manual default + ImageGenerationDialog com prompt editável. Decisão: adicionar **geração individual em slide vazio** (botão "Gerar imagem" em slide sem `image_versions`) ao escopo do PR3 em vez de espalhar pelo C6. Razão: virar módulo coeso em vez de features parciais espalhadas entre PRs. C6 fica focado em "regenerar/restaurar versão" (slide JÁ tem imagem); PR3 cobre "criar imagem sob demanda" + opções avançadas.

**5. Bug visual cover-numeric levantado mas NÃO corrigido na sessão.**

Diagnóstico fechado: `SlideRenderer.tsx:143` instancia `<CoverNumeric>` sem `imageUrl={img}`, mas o template renderiza trans inferior 480px se receber. Inconsistência colateral em `Editor.tsx:777` (`'cover-numeric'` em `noImage` do `needsImage`). Pedro decide entre 3 opções de fix (mínimo / +tech debt / outro) na próxima sessão.

## [05/05 noite] Edge function `generate-social-copy` — split de embedded join

**Contexto:** após o C2/C3 do PR2, geração de carrossel novo retornava 404 `POST_NOT_FOUND` apesar do post existir no banco. ULTRATHINK pedido pelo Pedro com evidência dos DevTools.

**Root cause:** a migration `20260505180000_social_palette_key_decouple` dropou a FK `social_posts_palette_key_fkey`. Edge function `generate-social-copy` linha 483 fazia `select('*, template:social_templates(*), palette:social_palettes(*)')` — PostgREST exige FK pra resolver embedded join. Sem FK, retorna `PGRST200`, edge function trata como `postError` e dispara 404 `POST_NOT_FOUND` (engana o diagnóstico).

**Decisão (opção A das 3 levantadas):** reescrever a query em 2 partes — `social_posts` com template embedded (FK existe), e palette por query separada via `key`. Coerente com o desenho do `palette_key_decouple` (frontend é fonte da verdade dos presets). Não restaurar a FK (opção B descartada — re-acoplaria à tabela legacy). Não usar FK NOT VALID/DEFERRABLE (opção C descartada — complexidade extra).

**Aplicado:** edge function `generate-social-copy` v11 deployada. `social-generate-image` JÁ estava no padrão correto (linhas 228-232, 278-282). Outras edge functions (`social-render-slide`, `social-render-carousel`, `social-export`) não tocam em `social_palettes`.

**Owner:** CC local via MCP Supabase deploy.

**Contexto:** PR2 do Social Studio (cor por elemento + versionamento de imagens). Plano em 6 commits incrementais. 3 decisões críticas alinhadas com Pedro antes de codar.

## [05/05] Canggu — root cause da regressão dupla (filtro Plataforma sumiu + Ana parou de classificar origem)

**Sintoma reportado pelo Pedro:** acessou `https://canguu-sigma.vercel.app/conversations` e (1) viu apenas 3 filtros (Status, Categoria, Sentimento) — o filtro de Plataforma implementado no commit `b2c6d0f` em 30/04 não estava visível; (2) percebeu que a Ana não estava mais mandando a mensagem numerada de classificação de origem nas conversas novas no WhatsApp.

**Diagnóstico:**
- **Filtro de Plataforma:** código presente no bundle deployado. `curl https://canguu-sigma.vercel.app/assets/index-B57kBlxQ.js | grep -E "Plataforma|Site Budamix"` confirmou que o JS em produção contém as strings. Componente `ConversationList.tsx:95-106` renderiza o `<Select>` incondicionalmente, sem feature flag. **Cache CDN do Vercel estava com `age: 393877s` (4.5 dias)** desde o último deploy em 30/04 18:09 UTC. Não era regressão de código.
- **Origin poll:** edge function `webhook-whatsapp` versão 28 deployada em **30/04 13:29 BRT**. Mas o commit `604b2e3` ("numbered-text origin poll + null source default") é de **30/04 17:28 BRT**, ~4h depois. Em produção rodava a versão anterior (`b2c6d0f`) que usava `sendList()` — Evolution + Baileys droppam silenciosamente esse tipo, ou mostram "aguardando mensagem" no celular do cliente. Numbered-text só foi pra produção agora.

**Resolução em 05/05 ~08:30 BRT:**
1. `supabase functions deploy webhook-whatsapp --project-ref jpacmloqsfiebvagfomt` → versão 29, com numbered-text origin poll.
2. `vercel redeploy <last-prod-deploy-id> --target=production` → cache CDN invalidado (`age: 0`, `last-modified` atualizado pra 05/05).

**Lição operacional:** edge functions Supabase **não auto-deployam** via push pro GitHub. Cada commit que toca `supabase/functions/*` precisa de `supabase functions deploy <name>` manual, **um por função**. Isso cria risco constante de drift entre repo (single source of truth declarada) e produção. Esse é o segundo evento documentado de drift desse tipo (primeiro foi tangencial na maratona de 30/04).

**Mitigação proposta para o roadmap Canggu (B5 — CI/CD):** GitHub Action no repo `PHPB2025K/canguu` que rode `supabase functions deploy --project-ref <ref>` automático no push pra main, idealmente filtrando por path (`supabase/functions/**`). Resolve a classe inteira de drift e remove necessidade de cada um lembrar de deployar manualmente.
