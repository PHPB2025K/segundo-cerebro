# Decisões — Junho 2026

## [03/06] Klap — dedup massivo de product_images (424 → 169)

**Contexto:** carrosséis das PDPs estavam carregando até 21 fotos por produto, várias visualmente idênticas (mesma foto Amazon SP-API em 3 resoluções: large/medium/`_SL75_` thumbnail).
**Decisão:** rodar pipeline de dedup em duas passadas — (1) URL base_id Amazon ignorando sufixo de tamanho removeu 108 thumbnails; (2) perceptual hash (pHash, threshold ≤ 5 Hamming) detectou outras 147 duplicatas entre base_ids Amazon diferentes (`7XXXX` vs `4XXXX` = mesma foto, IDs distintos).
**Resultado:** 255 fotos removidas, cada produto fica com 5–8 fotos únicas no carrossel. Mudança imediata em produção (sem deploy — só Supabase).
**Owner:** Pedro + Claude
**Status:** concluído

## [04/06] Integração Facebook Page Budamix via Graph API — token não-expirante + CLI local

**Contexto:** Pedro queria pedir edições na página Budamix do Facebook (capa, foto perfil, bio, etc) direto pelo Claude Code sem automação recorrente.

**Stack escolhido:** Facebook Graph API (v23.0) via curl/Bash + Page Access Token não-expirante armazenado em `~/.config/budamix-page/token` (perm 600). Wrapper CLI em `~/bin/budamix-page/fb-page` com subcomandos: info, about, update-about, update-description, upload-profile, upload-cover, get-cover, raw.

**Setup (~45min, com pivots):**
- Token derivado do app KOBE.OPENCLAW (`1486886096369858`) — Pedro inicialmente lembrava `3582660648568553` mas era app errado; confirmado via `/debug_token`
- App Secret correto: `9b7300ac211e368ff5f3ebba48a0f944` (do app `1486886096369858`)
- Fluxo: short user token (Graph Explorer) → long-lived (60d) via exchange → page token (não-expirante) via `GET /{page_id}?fields=access_token`
- Quirk: `/me/accounts` retornou `[]` mesmo com permissões corretas. Workaround: GET direto na página com user long-lived retorna o page token

**Edições aplicadas:**
- Capa: composição da MOBILE.png escalada 1640×924 (decisão "capa principal = mobile"; descartada opção de composição híbrida com DESKTOP.png por duplicação visual)
- Foto de perfil: arquivo final `Design sem nome (18).png` (logo Budamix® centralizado, ocupa ~85% do canvas, fundo Deep Teal)
- Bio (campo about): *"Vidro borossilicato para quem leva organização da cozinha a sério. Marmita fitness, meal prep e geladeira em ordem."* — opção 2A escolhida entre 5 variações sem usar "premium"

**Limitações encontradas:**
- **Vanity URL (username) bloqueado pela API:** `(#3) Application does not have the capability`. Requer aprovação "Page Public Content Access" da Meta. Pedro tentou configurar manualmente via Meta Business Suite mas a nova UI escondeu o campo; decidiu seguir com a URL canônica `https://www.facebook.com/106066888942641` por enquanto.
- **Confusão de URL:** Pedro tinha em mente `facebook.com/profile.php?id=100086248724676` — esse formato existe paralelo mas aponta pra mesma página `106066888942641` (sem 2ª página duplicada).
- **Capa não suporta variantes desktop/mobile** — Facebook tem 1 slot único; design 1640×924 com safe zone central é a melhor estratégia.

**Risco de segurança:** App Secret + user tokens iniciais ficaram no histórico do chat. App KOBE.OPENCLAW é compartilhado com o OpenClaw na VPS. Recomendação pendente: rotacionar App Secret no Meta Developers + atualizar OpenClaw.

**Owner:** Pedro + Claude
**Status:** concluído (page token salvo, 3 edições aplicadas)
**Próximos:** migração futura pra VPS / Kobe quando fizer sentido automatizar posts. Documentação em [[knowledge/concepts/facebook-page-api-budamix]].
